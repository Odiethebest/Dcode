> 给两位队友的项目启动说明。用最短的篇幅讲清楚：**我们要做什么、为什么、怎么走、你管哪一段、第一周先动哪里**。
>
> 这份文档读 10 分钟，[PLAN.md](PLAN.md) + [DESIGN.md](DESIGN.md) 读 1–2 小时——这份是入门，那两份是工作手册。

## 1. 一句话讲清楚我们要干啥

做一个**面向新人 onboarding 的代码理解平台**，叫 **Dcode**。

想象你刚加入一个几万行的开源项目，你最想问的是：
- "鉴权这功能到底怎么实现？走哪几个文件？"
- "这个函数有谁在调用？"
- "整个项目的请求处理流程是啥样的？"

现在的工具——GitHub Search 只能字面搜，Cursor 那种 RAG 只能做文本相似度——**回答的都不是这种"关系"问题**。

我们做的就是：**用户提交一个 GitHub repo，系统异步建一个『语义索引 + 代码调用图』双重索引，然后用一个多工具 Agent 用自然语言回答跨文件、架构层面的代码问题，并给出可跳转、可验证的代码引用。**

---

## 2. 为啥做这个（核心论点）

这是整个项目存在的理由，全部 eval 围绕它转。**记住这句话**：

> **传统关键字搜索（GitHub Search）和扁平向量 RAG，本质都解决"相似度"问题；而新人最想知道的"调用逻辑、架构层次、上下文依赖"，本质是"结构关系"问题。**
>
> **所以：结构感知索引（向量 + 代码图）+ Agent 多步工具调用，在需要跨文件推理的代码理解任务上，能显著优于扁平向量 RAG 和关键字搜索。**

这条假设**是可证伪的**——我们要用一个完整的 baseline 阶梯（GitHub Search → BM25 → Vanilla RAG → Hybrid RAG → 我们的完整系统）跑相同的题，看每一级能赢多少。

**🚨 重要纪律**：如果跑完发现我们没赢，**就如实写"假设未被支持"**，不调阈值、不改题、不挑数据。这是学术诚信，也是这个项目区别于"又一个 RAG demo"的根本。

## 3. 四周大致怎么走

| 周次 | 主线 | 周末应该能交付的东西 |
|---|---|---|
| W1 | **数据通路打通** | 索引管线能把一个真实 repo 从头到尾灌进数据库 |
| W2 | **端到端可问答** | 能对至少 5 个示例问题给出带引用的答案 |
| W3 | **评测可复现** | B0–B4 五个 baseline 都在问题集上跑完一轮 |
| W4 | **上线 + 收尾** | 部署到 `dcode.odieyang.com`、出最终结果、写报告 |

**每周交付原则**：纵向切片 MVP 优先——每周末必须有一个能跑通的东西，**宁可粗糙也不要"等我下周完美再交"**。

### 🟩 P3（待补名字）— 评测 & 前端

## 5. 三个**必须**钉死的接口（一旦冻结，谁都不许偷改）

| 同步点 | 截止时间 | 涉及方 | 冻结内容 |
|---|---|---|---|
| **S-1** | W1 周末 | Odie + P2 | 数据模型（4 张表：repos / chunks / symbols / edges）+ 索引 API |
| **S-2** | W2 周中 | Odie + P2 | 检索 API（`search` + 4 个图查询：`find_definition` / `find_references` / `get_dependencies` / `get_file_outline`） |
| **S-3** | W2 周末 | Odie + P3 | Agent 的 SSE 输出格式（7 类事件：`thought` / `tool_call` / `tool_result` / `citation` / `partial_answer` / `final_answer` / `error`） |

**为什么这么硬性？** 我们 3 个人并行干，一个接口动了下游所有人重构。任何接口改动**走 PR + 接口拥有者签**，**不要在群里一句话改**。详细字段定义看 [DESIGN.md §4](DESIGN.md)。

## 6. 几条不要踩的坑

- **不要做范围外的事**。多语言支持、IDE 插件、代码生成、配置文件解析、实时增量索引——**全砍**。看到自己想做了就停下来。
- **不要追求 UI 精致度超过演示需求**。能展示"完整系统 vs 裸 RAG"的对比就够，别陷进去调样式。
- **不要在代码里硬编码**：embedding 模型名、维度、reranker 端点、judge 模型、域名——**全走环境变量**。我们有 5 个 Open Decision（OD-1 到 OD-5）都没定，骨架阶段必须留好占位。
- **不要"等模块完美再交"**。每周末一个 MVP 纵向切片，粗糙没关系。
- **不要随便加依赖**。pgvector + Redis + RabbitMQ 已经够多了，加新东西先在群里讨论。
- **AST 切块不要用定长滑窗**。一定按函数 / 类边界切——否则 import 上下文丢了，检索结果脱离调用语境就废了（DESIGN 决策 **D-2.1.1**，硬约束）。
- **Groundedness 校验不可禁用**。Agent 答案里编造一个不存在的函数是项目级致命缺陷，**必须程序化校验后才返回**（**D-2.3.1**）。

## 8. 我们怎么算"赢"（验收指标）

跑完最后一轮 eval，看下面三个数字：

| 指标 | 目标 |
|---|---|
| **Retrieval**（Recall@k / MRR / nDCG） | 完整系统比每一级基线都强，**跨文件 / 架构级**题上差距显著 |
| **Pairwise Win-Rate**（我们 vs 裸 RAG） | **> 60%** |
| **Groundedness**（引用是否真实存在） | **≥ 95%** |

三个都达标 → H1 被支持，项目成功。
任一硬指标没达标 → 如实记录"假设未被支持"，分析为啥不行。
