## 1. 项目目标

项目在两个维度上同时设定目标：学术维度（验证假设、产出可复现的评测结果）与工程维度（构建符合生产级标准的后端基础设施）。两个维度的优先级关系在 §1.3 明确定义。

### 1.1 学术目标（首要）

| 编号 | 目标 |
|---|---|
| O1 | 在受控评测集上量化验证核心假设 H1（见 [README §核心假设](../README.md#核心假设-h1)） |
| O2 | 建立可复现的评测协议（问题集、基线阶梯、指标体系） |
| O3 | 提供按问题类型分层的差异化分析（单文件 / 跨文件 / 架构级） |

### 1.2 工程目标（次要）

| 编号 | 目标 |
|---|---|
| E1 | 实现支持异步索引、多租户隔离、缓存优化的后端服务平台 |
| E2 | 实现答案级别的程序化引用校验（groundedness guardrail） |
| E3 | 完成端到端部署与演示界面 |

### 1.3 优先级声明

学术目标 O1–O3 的达成是**项目成立的必要条件**：若 H1 未被验证，工程目标的成果将失去叙事载体。工程目标 E1–E3 不参与假设的证明，但构成系统的**可演示性**与**生产级可信度**。

时间紧张时按 §1.1 优先于 §1.2 的顺序保留范围；详见 §4 降级路径。

## 2. 项目范围

### 2.1 范围内 (In Scope)

- **支持语言**：Python（单一语言，依据生态成熟度选定）
- **目标仓库**：1–3 个中等规模主流开源 Python 项目（候选：`requests` / `flask` / `fastapi`）
- **核心能力**：异步索引管线、混合检索（dense + sparse + rerank）、代码结构图、多工具 ReAct Agent、端到端评测、Web 演示界面

### 2.2 范围外 (Out of Scope)

| 项 | 决策依据 |
|---|---|
| 多语言支持 | 不影响 H1 验证，工具链复杂度高 |
| 通用配置文件 / 构建系统解析 | 边际价值低 |
| 代码图大规模可视化 | 不影响验收指标 |
| 实时增量索引 | 首版全量重建即可满足评测 |
| 代码生成 / 修改 / 重构 | 偏离项目定位 |
| IDE 插件形态 | 偏离项目定位 |

### 2.3 非目标 (Non-Goals)

明确声明本项目**不追求**的方向，以约束范围漂移：

- 本项目**不是**代码生成或修改工具，不进行任何写入仓库的操作；
- 本项目**不是**通用聊天机器人，Agent 仅作为假设 H1 的演示前端而非产品形态；
- 本项目**不追求**支持任意仓库的开箱即用，目标仓库在评测前预先选定并完成索引；
- 本项目**不追求** UI 精致度优于演示需求，前端实现目标是清晰承载 §3.2 定义的对比场景。

## 3. 验收标准

项目验收以**定量指标**与**定性产出**双重达成为依据。

### 3.1 定量指标

| 指标类别 | 指标 | 目标 |
|---|---|---|
| Retrieval | Recall@k、MRR、nDCG | 完整系统相对每一级基线均有可观测改进，且改进在"跨文件 / 架构级"子集上具有统计显著性 |
| Agent 答案质量 | LLM-as-Judge 评分、Pairwise Win-Rate | 完整系统对裸 RAG 基线的 Win-Rate 高于 60%（具体阈值依问题集规模调整） |
| 答案可信度 | Groundedness（引用符号 / 路径在仓库内可程序化验证存在的比例） | ≥ 95% |

基线阶梯依次为：GitHub Search → BM25 → Vanilla Dense RAG → Hybrid RAG → Hybrid + 结构图（完整系统）。每一级的增量贡献需独立报告。详见 [DESIGN.md §2.4 评测子系统](DESIGN.md#24-评测子系统-evaluation-harness)与 [§4.4 评测协议](DESIGN.md#44-评测协议)。

### 3.2 定性产出

- **对比演示场景**：针对同一自然语言查询，并列呈现完整系统答案（跨文件、带可跳转引用、引用经程序化校验）与裸 RAG 基线答案，作为项目代表性产出；
- **评测报告**：包含问题集构造方法、问题类型分类（taxonomy）、各基线在各类问题上的指标对比，与对假设 H1 的最终判定。

## 4. 风险驱动的优先级与降级路径

在资源或时间约束下，按以下优先级顺序保留功能范围。低优先级项可在不影响假设验证的前提下削减。

| 优先级 | 内容 | 对应目标 | 削减后果 |
|---|---|---|---|
| P0 | 结构感知检索（向量 + 代码图） | O1 | 假设无法验证，项目失败 |
| P0 | 评测协议与基线阶梯实现 | O2、O3 | 假设无法量化，项目失败 |
| P1 | 程序化 groundedness 校验 | E2、§3.1 第三项 | 答案可信度无法保证，演示受损 |
| P1 | 对比演示场景（UI 最小集） | §3.2 第一项 | 项目缺乏可视化呈现 |
| P2 | 异步索引管线、多租户、缓存 | E1 | 假设验证不受影响；工程叙事价值下降 |
| P3 | UI 精致度、代码图可视化、性能调优 | — | 无实质影响 |

**关键判断**：P2 项（异步管线与基础设施）**不参与假设 H1 的验证**。理论上可使用同步脚本完成索引并通过全部评测。其价值体现在系统的工程完整度与可扩展性叙事，应在 P0 与 P1 项稳定后投入。

### 4.1 里程碑级降级动作

| 触发条件 | 动作 |
|---|---|
| W1 延期 | 削减多租户隔离的实现复杂度，简化为单租户演示部署 |
| W2 延期 | 削减工具清单中的 `list_directory`、`grep`，保留与 H1 验证直接相关的核心工具 |
| W3 延期 | 削减前端 P1/P2 功能；削减 B0、B1 基线的运行（保留 B2、B3 对 B4 的对照） |

## 5. 约束与假设

### 5.1 约束

- **时间窗口**：项目执行窗口为 4 周；与其他并行任务（课程、求职准备）共享资源。
- **团队规模**：3 人；分工见 §6。
- **预算**：避免高频商业 API 调用；embedding 优先使用自托管开源模型。

### 5.2 假设

- 目标开源仓库的代码质量与文档完备性，足以支持人工与半自动方式构造可靠的评测 ground truth；
- 选定的开源代码 embedding 模型在 Python 语料上的表现，足以支撑超越通用 embedding 的检索效果；
- LLM-as-Judge 在配合 pairwise 比较与 rubric 引导下，能产生与人工判断相关性可接受的评分。

上述假设在项目第 1 周完成验证；任一假设证伪将触发方案调整（见 §8 风险登记 R-1、R-2）。

## 7. 里程碑与排期

### 7.1 总体排期

| 周次 | 里程碑 | Exit Criteria |
|---|---|---|
| W1 | **M1 — 数据通路打通** | 目标仓库经索引管线完整写入数据库；DESIGN §3 数据模型冻结；§4.1 接口实现并通过冒烟测试 |
| W2 | **M2 — 端到端可问答** | DESIGN §2.2、§2.3 完成；可对至少 5 个示例问题端到端产出带引用的答案；§4.2、§4.3 冻结 |
| W3 | **M3 — 评测可复现** | B0–B4 全部 Baseline 在问题集上完成至少一轮跑分；指标产出符合 §4.4 格式；UI 主路径可用 |
| W4 | **M4 — 上线与交付** | 部署至 `dcode.odieyang.com`；最终评测结果产出；技术报告 / README / 架构图完成 |

**每周交付原则**：MVP 优先，每周一个能跑通的纵向切片；不追求模块"完美"再集成。

### 7.2 准入与准出

每个里程碑必须满足前一里程碑的 Exit Criteria 方可启动；任一 Exit Criteria 未达成不得跨周。

### 7.3 并行度安排

- **W1**：Odie heads-down 索引管线 A；Yuxin 搭基建 + 与 Odie 共定数据模型；Yufan 独立构造问题集 + 评测脚手架（最可并行）；
- **W2**：Odie 搭 Agent C（基于 Yuxin 的检索 API）；Yuxin 完成 hybrid + 图查询；Yufan 对着 mock agent 接口搭 UI 壳；
- **W3**：Yufan 主场，跑全量 baseline + UI 收尾；Odie + Yuxin 提供性能与稳定性保障；
- **W4**：集成、部署、最终结果产出、文档收尾。

## 10. 术语表

| 术语 | 定义 |
|---|---|
| **结构感知索引** | 在语义向量索引之外，附加由静态分析（定义、引用、依赖）构建的代码调用图 |
| **基线阶梯** | 由弱到强的多个对照系统序列，用于隔离每一项技术改进的边际贡献 |
| **Groundedness** | 生成答案中所引用的代码实体（符号、路径、行号）在源仓库内可被程序化验证为真实存在的比例 |
| **Taxonomy（问题分类）** | 评测问题集按所需推理范围划分为单文件事实型 / 跨文件结构型 / 架构理解型三类，用于差异化分析 |
| **Hybrid Retrieval** | dense（向量）+ sparse（BM25）双路召回后经 reranker 融合的检索方案 |
